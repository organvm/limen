## Root Cause
The OpenCode agent operates at the product surface and governance interface layer, tasked with deploying cloud infrastructure and coordinating organizational artifacts. However, productive board spend for capacity-fill requires *actual code changes* to be committed to *owner repositories*. The opencode lane's domain is deployment and configuration, not core revenue/model development work.

This creates a structural misalignment:
- **Domain Mismatch**: opencode < deploys / configures > vs. **required work**: core code changes / revenue generation
- **Trust Boundary**: productive board spend requires owner repository access and trusted commit authority not granted to the opencode subagent
- **No Work Range**: No tasks in the current board that fall within opencode's lane capabilities (configure, deploy, product surface)

## Next Steps
