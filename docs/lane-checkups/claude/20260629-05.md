# Claude Lane Checkup 20260629-05

## Blocker
The `claude` lane is blocked from dispatch because its required fleet credentials (`LIMEN_CLAUDE_API_KEY` or `LIMEN_CLAUDE_AUTH_TOKEN`) are missing from the *running* dispatch environment. `cli/src/limen/dispatch.py`'s `_creds_lanes_down()` explicitly gates headless dispatches unless one of these fleet-mode credentials is set in `os.environ`, to prevent hanging or wasting attempts on an unauthenticated CLI.

## Evidence
1. `dispatch-health.md` shows: `claude: lane is down by the live dispatch gate`
2. The credential gateway (`cli/src/limen/dispatch.py:133-134`) checks runtime `os.environ.get("LIMEN_CLAUDE_AUTH_TOKEN")` and `os.environ.get("LIMEN_CLAUDE_API_KEY")`
3. The running heartbeat daemon was started before the credential was written to `~/.limen.env`. It loads env files at startup, so the new credential is not yet active in the dispatch process.

## Resolution Command
The next step is human-gated. You must restart the heartbeat daemon to reload `~/.limen.env`:

```bash
launchctl kickstart -k gui/$(id -u)/com.limen.heartbeat
```

Then re-verify the lane is back online using:
```bash
python3 scripts/dispatch-health.py --write --probe-async
python3 scripts/capacity-fill-ledger.py --write
```
